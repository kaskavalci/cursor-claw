# CLI subprocess → cursor-sdk migration

**Date:** 2026-07-05  
**Author:** Composer

The Telegram bot no longer shells out to `cursor agent`. It uses **cursor-sdk** (`Agent.create`, `Agent.resume`, streaming `run.messages()`).

## What changed

| Before | After |
|--------|-------|
| `cursor agent` CLI on PATH | `CURSOR_API_KEY` in `telegram-bot/config` |
| Session ids: CLI UUIDs | Session ids: `agent-…` |
| `python3 telegram-bot/agent_bot.py` | `uv sync` then `make bot-run` or `.venv/bin/python` |
| `/summarize` via ask mode | `/summarize` via SDK plan mode |
| `/model` via `agent models` | `/model` via `Cursor.models.list()` |

Legacy CLI UUIDs in `.cursor_agent_sessions.json` are **dropped on load**. They cannot be resumed.

## Upgrade steps

1. **Install deps** (repo root):

   ```bash
   uv sync
   ```

2. **Config** — add to `telegram-bot/config`:

   ```bash
   CURSOR_API_KEY=your_key_from_cursor_dashboard
   ```

   Optional: `CURSOR_AGENT_TIMEOUT=600` (seconds; `0` = no limit).

3. **Deploy / restart** — e.g. `make deploy` or restart `telegram-agent-bot.service` (units use `.venv/bin/python`).

4. **Start fresh session** — send `/new` once on Telegram. Old CLI chats remain in the registry file but are ignored; use `/chats` and pick a listed `agent-…` session or `/new`.

5. **Smoke test** — `/status`, `/model`, a short prompt, `/summarize`.

## Rollback

Check out the previous commit and restart the bot with the CLI-based code. SDK session ids will not work with the old subprocess path.

## Files added

- `pyproject.toml`, `uv.lock` — `cursor-sdk` dependency
- `telegram-bot/agent_pool.py` — in-process agent handle cache
- `telegram-bot/agent_runner.py` — SDK runs and Telegram delivery
- `telegram-bot/config_loader.py` — shared config loading
